## 3. What changed in the provider journey

| Old (in `outputs/provider-journey.html`) | New (in `docs/journeys/provider-journey-v2.html`) |
|---|---|
| Provider opens job offer â†’ views in PWA â†’ unlock + accept in one step | Provider sees safe preview inline in WhatsApp; WhatsApp completes the entire core lifecycle |
| Credit deducted on the first provider's acceptance (legacy `acceptLead` path) | Credit deducted **only** on selected-provider final acceptance after customer selection |
| No structured way for a provider to express interest with rate / ETA | Multi-step `INTERESTED` capture in WhatsApp: call-out fee â†’ ETA â†’ rate / negotiable / note (free, no credit) |
| `LeadUnlock` and `Match` created at first-accept | `LeadUnlock`, `Match`, `Quote`, `Booking`, `Job`, `JobStatusEvent`, `AuditLog` all created in one atomic transaction inside `acceptSelectedProviderJob` |
| Unlock flow described as "Unlock & Accept" | Two visible button labels: `confirm_accept:<leadId>` and `confirm_decline:<leadId>` after customer selection |
| Provider PWA implicitly required for richer screens | Provider PWA explicitly **optional**; every transactional action has a working WhatsApp path (apply, status, balance, interest, accept selected, full details, arrival, on the way, arrived, start, complete, completion note + photo) |
| Ambiguous handling of `MORE_INFO_REQUIRED` | Explicit return path via `resumeMoreInfoApplication` â€” provider replies in free-text, status returns to `PENDING`, audited |
| Phone numbers in logs not masked | All identified high-traffic phone logs masked via `maskPhone` (`0XX****XXX`) |
| Token routes (`/provider/handoff|lead|job/:token`) treated as plain redirects | Routed through `resolveProviderPwaHandoffPath` for state-aware resolution |

The new diagram now has eleven readable subgraphs: WhatsApp entry & identity â†’ onboarding â†’ review & approval â†’ operations menu â†’ opportunity preview â†’ interest response â†’ customer selection notification â†’ final acceptance â†’ arrival/execution â†’ optional PWA surfaces â†’ exception/recovery.

## 5. What changed in the shortlist / acceptance / credit flow

The matching/shortlist/credit flow is now its own first-class diagram (`matching-shortlist-credit-flow-v2.html`) because the credit timing and privacy boundary needed a single unambiguous place to be expressed. Key points it now makes explicit:

- **Credit is deducted exactly once**, on the selected provider's final acceptance via `unlockLeadForProviderInTransaction` â€” the single shared ledger entrypoint.
- Credit is **not** deducted on:
  - Lead invite dispatch / preview
  - Provider `INTERESTED` response (with rate + ETA)
  - Provider `NOT_INTERESTED` / no reply
  - Shortlist generation (`SHORTLIST_READY`)
  - Customer selection (`PROVIDER_CONFIRMATION_PENDING`)
  - Selected-provider WhatsApp notification
  - Selected-provider `confirm_decline` (request returns to `SHORTLIST_READY`)
  - Re-accept by same provider (idempotent â€” `alreadyUnlocked: true`)
- **Privacy unlock** is co-incident with the credit deduction â€” same atomic transaction. Before unlock, provider sees only safe preview fields; after unlock, the accepted provider sees customer name, phone, full address, unit/complex, structured `accessNotes`, full description, all photos, preferred time, and job reference.
- The diagram includes a printed reference table at the bottom for credit/privacy outcomes per event.
- It walks through the JobRequest state machine `OPEN` â†’ `MATCHING` â†’ `SHORTLIST_READY` â†’ `PROVIDER_CONFIRMATION_PENDING` â†’ `ASSIGNED/SCHEDULED` and the LeadInvite logical lifecycle (`SENT` â†’ `VIEWED` â†’ logical `INTERESTED` â†’ logical `SHORTLISTED` â†’ logical `CUSTOMER_SELECTED` â†’ `ACCEPTED`, plus terminal `DECLINED`, `EXPIRED`, logical `SUPERSEDED`, logical `CANCELLED`).

## 6. Privacy boundary â€” explicit field map

**Hidden BEFORE selected-provider acceptance (preview / shortlist / pre-confirm):**
- Customer phone, customer email
- Exact street address, house number, unit number, complex name, complex access details
- Structured `Address.accessNotes`
- GPS coordinates (lat/lng)
- Postal code
- Provider phone (always masked from other parties)
- Provider private address, ID/passport, private documents, reference contact details, admin notes (always hidden from customers)
- Attachments flagged `safeForPreview: false`
- Customer free-text description is truncated via `previewNotes`

**Visible AFTER selected-provider acceptance (only to accepted provider):**
- Customer full name, customer mobile number
- Full service address, unit / complex details
- Structured `accessNotes`
- Full job description, all attached photos, preferred time, job reference

**Token / URL rules:**
- Customer ticket tokens are scoped to one request and expire.
- Provider lead tokens are scoped per-provider per-lead.
- Production WhatsApp URLs use `getPublicAppUrl`, which fails closed in production (rejects `localhost` / `127.0.0.1`).

## 8. Assumptions made

1. **Target state both flags ON.** The diagrams reflect the target state with feature flags `qualified_shortlist.dispatch_v2` and `qualified_shortlist.auto_trigger` both ON. In the as-shipped pilot config they default OFF, in which case providers responding via WhatsApp still go through the legacy `accept:<holdId>` / `decline:<holdId>` path. This is flagged but not drawn â€” drawing both states would have been confusing. The legacy fallback is a code path, not a target user journey.
2. **Call-out fee becomes the approved Quote at acceptance.** `acceptSelectedProviderJob` synthesises an `APPROVED` `Quote` at the provider's call-out fee at acceptance time. Both Codex and Claude reviews flag this as a product decision (it short-circuits a separate quote-after-inspection step). Diagrams treat the call-out-fee-as-quote as the shipped behaviour.
3. **Effective lead lifecycle, not raw enum.** The Prisma `LeadStatus` enum has both legacy values (`SENT`, `VIEWED`, `ACCEPTED`, `DECLINED`, `EXPIRED`) and additive ones (`INTERESTED`, `SHORTLISTED`, `CUSTOMER_SELECTED`, `SUPERSEDED`, `CANCELLED`). Today's code does not yet write the additive values â€” effective state is inferred from timestamp columns and `ProviderShortlistItem` membership. Diagrams show the **logical** lifecycle and explicitly note that some values are timestamp-driven.
4. **PWA handoff trigger.** Docs describe handoff as the natural pattern but do not specify a precise trigger. The customer diagram treats handoff as triggered when (a) photos / structured address / urgency / preference / budget capture is needed, OR (b) the shortlist UI must be rendered.
5. **Threshold default.** `SHORTLIST_AUTO_TRIGGER_THRESHOLD` default of 2 is shown explicitly.
6. **Customer sign-off `PENDING_COMPLETION_CONFIRMATION` â†’ `COMPLETED`.** Drawn as TBD because it is "outside this Client PWA remediation scope" per the source docs.
7. **Provider WhatsApp commands.** ETA NLP forms (`HH:MM`, `today/tomorrow + window`, `in N hours`, `noon`, `later today`, etc.) shown per the third-pass remediation. If any of these were rolled back, diagrams will need a small refresh.
8. **No mandatory PWA paths for provider operational flow.** The provider diagram explicitly shows the PWA as optional throughout â€” no transactional step requires it.

## 9. Unresolved ambiguities
