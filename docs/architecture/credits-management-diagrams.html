💳 Plug A Pro — Credits Lifecycle

End-to-end flow: onboarding grants · top-up purchases · lead unlock debits · reconciliation

📅 2026-04-30 🔍 Source-code audit 📄 credits-management-review.md

Architecture verdict

The credits implementation is materially present and stronger than a shallow balance counter. A dedicated wallet module, immutable ledger rows, paid vs promo credit separation, transaction-scoped debits, and idempotency guards for grants/top-ups/unlocks are all in place.

The core top-up and lead-unlock transaction paths are robust. Reconciliation, reversals, test-cohort isolation, and operational recovery are partial. No P0 production-money issue was found. Safe for controlled pilot with manual ops oversight.

P0 clear — no money leak found in core path P1 — no automated reconciliation report P1 — test cohort ledger not isolated P1 — reversal & expiry wired in schema only P1 — suspend/reactivate ledger check constraint conflict P1 — no balance drift repair command

Lead Unlock

Credit Deduction — Lead Unlock & Accept

One atomic transaction: eligibility check → unlock + ledger debit → assignment + match → notifications. All three channels (WhatsApp, PWA signed link, Worker Portal) funnel through the same acceptLead → acceptAssignmentOffer → unlockLeadForProviderInTransaction stack. Promo credits are consumed before paid credits.

Reconciliation

Reconciliation & Audit

Current reconciliation is entirely query-based and manual. No automated finance job or exception report was found. The data model supports full reconciliation queries; the ops tooling to run them does not yet exist.

credits-reconciliation-flow.mmd
Grant checkApproved providers → ProviderPromoAward → PROMO_CREDIT entries (at most 1 MOBILE_VERIFIED per provider)
Top-up checkPaymentIntent CREDITED → TOPUP_CREDIT entry → creditedLedgerEntryId match
Unlock checkLeadUnlock → LEAD_UNLOCK_DEBIT entries (should be exactly 1 per unlock)
Balance checkProviderWallet cached balance = latest ledger balanceAfter snapshot
Dispute checkApproved LeadUnlockDispute → LEAD_REFUND_CREDIT entries present
Admin adj checkADMIN_ADJUSTMENT entries have non-empty reason + AdminAuditEvent

Gap — no automated reconciliation exists today All reconciliation checks listed above must be run as manual SQL queries. There is no cron job, admin report page, or finance export that surfaces exceptions automatically. This is acceptable for a controlled pilot but must be addressed before scale.

Recommended first reconciliation query: SELECT p.id, p.name, w.paidCreditBalance + w.promoCreditBalance AS cached, MAX(l.balanceAfter) AS ledger_last FROM providers p JOIN provider_wallets w ON w.providerId = p.id JOIN wallet_ledger_entries l ON l.providerId = p.id GROUP BY p.id, p.name, w.paidCreditBalance, w.promoCreditBalance HAVING cached != MAX(l.balanceAfter)
